**日期**: 2026-05-08  
**状态**: 待 Review

---

## 1. 项目概述

一个移动端 H5 网页工具，解决高德地图在路线规划中的两个痛点：

1. **多方式对比缺失** — 高德在规划市内路线时，通常只默认展示一种或两种方式（如仅公交/地铁），无法一眼对比所有交通方式的总耗时和费用。
2. **途径点不支持公共交通** — 高德途径点功能主要针对驾车路线，公共交通（公交/地铁）不支持多段中途停靠规划。

此外，本工具还将**铁路出行**纳入整体规划，支持与 12306 车次数据结合，实现"市内交通 + 铁路 + 市内交通"的完整门到门路线规划。

### 典型使用场景

> 用户要从龙岗平湖站出发，先到深圳站，再到惠州北站。  
> 工具需要告诉用户：
> - 平湖站 → 深圳站：打车 45 分钟 ¥80 / 地铁 65 分钟 ¥8 / ...
> - 深圳站 → 惠州北站：高铁 G1234，30 分钟 ¥35
> - 整体最快方案：地铁 + 高铁 = 总计 1 小时 35 分钟，¥43

## 3. 核心模块设计

### 3.1 火车代理模块 (`TrainProxy`)

封装 12306 接口的所有脏活，对上层暴露干净的查询方法。

**职责**:
- **车站编码转换**: 12306 使用内部编码（如北京=`BJP`，上海=`SHH`），需从 `station_name.js` 解析映射表
- **请求伪装**: 模拟浏览器请求头（Referer、User-Agent、Cookie），绕过基础反爬
- **结果缓存**: 内存缓存（TTL 5 分钟），相同查询减少重复请求
- **容错降级**: 12306 接口偶发异常时捕获并返回空列表，不影响其他行程段

**暴露接口**:
```python
class TrainProxy:
    async def query_trains(
        self, 
        from_station: str,    # 中文站名，如"深圳"
        to_station: str,      # 中文站名，如"惠州北"
        date: str             # YYYY-MM-DD
    ) -> List[TrainInfo]
    
    async def query_transfers(
        self,
        from_station: str,
        to_station: str,
        date: str
    ) -> List[TransferInfo]
```

### 3.2 路线聚合模块 (`RouteAggregator`)

对单段行程，并行调用高德对应的路径规划 API，统一返回格式。

| 用户选择方式 | 高德 API | 说明 |
|-------------|---------|------|
| 公交/地铁 | 公交路径规划 | 包含换乘方案 |
| 驾车 | 驾车路径规划 | 含实时路况估算 |
| 骑行 | 骑行路径规划 | — |
| 步行 | 步行路径规划 | — |

**统一输出结构**:
```python
class RouteSegment:
    mode: str           # "transit" | "driving" | "bicycling" | "walking"
    duration: int       # 秒
    cost: float         # 元（驾车/打车时含估算油费/打车费）
    distance: int       # 米
    steps: List[Step]   # 详细步骤
```

**注意**: 高德 API 对"打车"没有单独接口，驾车路径 + 估算打车费公式（如 `distance * 单价 + 起步价`）来模拟。

### 3.3 组合优化模块 (`RouteOptimizer`)

给定 N 段行程，每段有 M 种可选方式，找出最优组合。

**算法**: 简单回溯枚举（DFS）
- 段数通常 2~4 段，每段方式 2~6 种，组合空间极小（最坏 < 1000 种），毫秒级求解
- 支持两种优化目标：总耗时最短（默认）、总费用最低

**衔接约束**:
- 若第 i 段终点为火车站，第 i+1 段为火车，需在第 i 段到达时间基础上 **+30 分钟进站预留时间**，再匹配火车发车时间
- 若火车到达后接市内交通，同样 +15 分钟出站预留

**输出**:
```python
class PlanResult:
    total_duration: int          # 总耗时（秒）
    total_cost: float            # 总费用（元）
    segments: List[PlanSegment]  # 各段详情
```

## 4. 数据流

```
用户输入行程（起点 → 途径点1 → ... → 终点）
        │
        ▼
前端 POST /api/plan
        │
        ▼
后端 ──► 并行查询每段的高德API（多种方式）
        │
        ├──► 若涉及火车站，额外查询12306车次/中转
        │
        ▼
    统一格式化所有结果
        │
        ▼
    组合优化（枚举 + 排序）
        │
        ▼
    返回 Top 3 方案
        │
        ▼
前端时间轴展示 + 支持手动调整某段方式
```

## 5. API 设计

### 5.1 路线规划接口

```http
POST /api/plan
```

**Request**:
```json
{
  "departure_time": "2026-05-09T08:00:00",
  "segments": [
    {
      "from": {"name": "龙岗平湖站", "lng": 114.123, "lat": 22.456},
      "to": {"name": "深圳站", "lng": 114.321, "lat": 22.654},
      "modes": ["transit", "driving", "bicycling"]
    },
    {
      "from": {"name": "深圳站", "lng": 114.321, "lat": 22.654},
      "to": {"name": "惠州北站", "lng": 114.567, "lat": 22.789},
      "modes": ["train"]
    }
  ],
  "optimize_by": "duration"
```

**Response**:

```json
{
  "plans": [

          "duration": 3900,

          "details": { ... }
```

### 5.2 POI 搜索接口（代理高德）

```http
GET /api/poi/search?keyword=深圳站&city=深圳
```

返回地点列表（名称、地址、经纬度），供前端输入框联想。

### 5.3 车站列表接口

```http
GET /api/train/stations
```

返回 12306 所有支持的车站名称列表，供前端判断某地点是否为火车站。

## 6. 前端交互设计

### 6.1 页面1：行程输入

- **起点/终点输入框**: 带 POI 搜索联想，调 `/api/poi/search`
- **途径点**: 起点和终点之间可插入/删除途径点，最多支持 3 个；支持拖拽排序
- **方式选择**: 一排可多选的标签按钮
  - 🚌 公交/地铁
  - 🚗 驾车（含打车估算）
  - 🚴 骑行
  - 🚶 步行
  - 🚄 火车
  - 默认全选
- **出发时间**: 日期时间选择器
- **底部**: 大号主按钮"规划路线"

### 6.2 页面2：结果展示

- **顶部 Tab**: "最快到达" / "最省费用" 切换，重新排序
- **方案卡片**（展示 Top 3）:
  - 头部：总耗时（如 2小时35分）、总费用（如 ¥86）
  - **纵向时间轴**：
    - 每段一个节点，用不同颜色/图标区分方式
    - 显示：出发时间 → 方式图标+名称 → 到达时间
    - 火车段额外显示车次号（如 `G1234`）和座位类型
  - **可展开详情**：点击某段展开具体步骤（如地铁换乘站、驾车路线、火车历时）
- **手动调整**：点击时间轴某段，底部弹出方式选择器，切换后实时重新计算总时间和总费用
- **底部**: "修改行程"返回输入页

### 6.3 异常状态

- 某段无可用方案（如某段无法骑行）：该段显示"无可行方案"，不参与组合
- 12306 查询失败：火车段显示"暂无可用车次，请手动查询"，其他段正常展示
- 网络错误：全局 Toast 提示

## 8. 外部依赖与数据源

### 8.1 高德地图 API

- **Key 申请**: 高德开放平台申请 Web 服务 Key
- **使用配额**: 个人开发者有免费日限额，初期足够
- **涉及接口**:
  - 地理编码 / 逆地理编码
  - POI 搜索
  - 路径规划（公交、驾车、骑行、步行）

### 8.2 12306 网页接口

- **数据来源**: `kyfw.12306.cn` 网页版查询接口
- **车站编码表**: `https://kyfw.12306.cn/otn/resources/js/framework/station_name.js`
- **车次查询**: `https://kyfw.12306.cn/otn/leftTicket/query`
- **中转查询**: `https://kyfw.12306.cn/otn/lcQuery/query`
- **风险**: 接口格式可能随时调整；高峰期可能限流或要求额外验证（如滑块）

## 9. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| 12306 接口变更或加反爬 | 火车查询功能失效 | 模块解耦，火车代理异常时不影响市内交通规划；预留降级提示 |
| 12306 限流/IP 封禁 | 请求被拒绝 | 后端加请求间隔（如 1 秒）、内存缓存、User-Agent 轮换 |
| 高德 API 配额耗尽 | 无法规划市内路线 | 监控用量，必要时申请提升配额 |
| CORS 问题 | 前端直接调 12306 被拒 | 已由后端代理解决 |
| 打车费用估算不准 | 用户决策偏差 | 明确标注"估算费用，以实际为准" |

## 10. 第一版范围（MVP）

为了快速验证核心功能，第一版先做以下范围：

- ✅ 支持 2 段行程（起点 → 终点，或 起点 → 途径点 → 终点）
- ✅ 市内交通方式：公交/地铁、驾车、骑行、步行
- ✅ 火车查询：直达车次（暂不做中转方案）
- ✅ 优化目标：总耗时最短
- ✅ 前端：输入页 + 结果页（时间轴展示）
- ❌ 后续版本：多途径点（3+）、火车中转、费用最低排序、手动调整方式、历史记录
